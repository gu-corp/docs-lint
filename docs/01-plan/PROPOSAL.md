# docs-lint プロジェクト提案書

**バージョン**: 1.0
**更新日**: 2025-12-10

## ビジョン

G.U.Corp の全プロジェクト（90以上のリポジトリ）において、ドキュメント品質を自動的に検証・統一し、AIと人間の両方が効率的に理解・活用できるドキュメント環境を実現する。

## 解決する課題

### 現状の問題

1. **ドキュメント構造の不統一**
   - リポジトリごとに異なるフォルダ構成
   - ファイル命名規則がバラバラ
   - 必須ドキュメントの欠如

2. **品質のばらつき**
   - バージョン情報の欠如
   - リンク切れ
   - 未解決の TODO/FIXME
   - 用語の不統一

3. **AIによる活用の困難さ**
   - 構造化されていないドキュメントはAIが解析しにくい
   - 要件とテストケースの対応関係が不明確

### 期待される効果

| 課題 | 解決策 | 効果 |
|------|--------|------|
| 構造の不統一 | 標準フォルダ構成の強制 | 全プロジェクトで一貫した構造 |
| 品質のばらつき | 自動リントチェック | CI/CDで品質を担保 |
| AI活用の困難さ | FR-XXX/TC-XXX形式の強制 | 要件追跡が容易に |

## ターゲットユーザー

| ユーザー | 用途 |
|---------|------|
| ソフトウェアエンジニア | プロジェクトのドキュメント品質維持 |
| テクニカルライター | ドキュメント作成時の品質チェック |
| プロジェクトマネージャー | ドキュメント完成度の確認 |
| AIアシスタント（Claude等） | 構造化されたドキュメントの解析・生成 |

## 主要機能

### 1. コンテンツチェック

- リンク切れ検出
- バージョン情報の有無
- 見出し階層の検証
- TODO/FIXME の検出
- コードブロックの言語指定チェック
- 用語統一の検証

### 2. 構造チェック

- 標準フォルダ構成の検証
- 必須ファイルの存在確認
- ファイル命名規則の検証
- 番号付けフォルダの連番チェック

### 3. 要件・テスト追跡

- FR-XXX形式の要件ID検出
- TC-XXX形式のテストケースID検出
- 要件カバレッジの検証

### 4. AI連携

- AI向けプロンプト生成（`--ai-prompt`）
- JSON形式の出力（`--json`）
- ドキュメント標準テンプレートの提供

## 成功指標（KPI）

| 指標 | 目標値 |
|------|--------|
| 導入リポジトリ数 | G.U.Corp全リポジトリ（90+） |
| lint通過率 | 95%以上 |
| ドキュメント構造統一率 | 100% |
| CI/CD統合率 | 80%以上 |

## スコープ

### In Scope（対象）

- Markdownファイルのリントチェック
- フォルダ構成の検証
- CLI / API インターフェース
- GitHub Actions統合
- 日本語・英語対応

### Out of Scope（対象外）

- Markdown以外のフォーマット（Word, PDF等）
- ドキュメントの自動生成（チェックのみ）
- GUI（CLIのみ提供）
